Context

What does a compiler actually do?

It turns source code that humans write into a structured form a machine can act on. We do this in four well-defined phases.

Each phase consumes the output of the previous one. If any phase rejects the input, the pipeline stops and reports a clear error.

Explain the pipeline in plain words: characters become tokens, tokens become a tree, the tree is checked for meaning, then it's lowered to a simpler instruction-like form (TAC). The optimizer is optional and toggleable with --no-opt.
Objective

What the spec asks for

Required

  • Variables & expressions (arithmetic, relational, logical)
  • Control structures: if, while, for
  • Functions including main and user-defined
  • Symbol table with global & local scope
  • Lexer, parser, semantic analyzer, IR (e.g. TAC)

Innovation (2 marks)

  • Arrays — declaration, indexed read & write
  • Constant folding — evaluate constants at compile time
  • Dead code elimination — drop unreachable instructions

Everything required is implemented. We chose two low-risk innovations that demo well and add real depth.

Design choices

  • One module per phase. Easy to reason about and to grade.
  • Pure stdlib. No third-party packages, runs anywhere with Python 3.10+.
  • Hand-written lexer & parser. No yacc / PLY — every line is ours and explainable.
  • Dataclass AST. Fewer keystrokes, free __repr__, easy to traverse.
  • Golden-file tests. Compile each program, diff the TAC. Catches regressions instantly.
Show the file tree and emphasise modularity — one module per phase. Mention the test harness runs every sample and diffs against expected output.
Phase 1 · Lexer

Source → Tokens

Source

int x = 3 + 4;

What it does

  • Reads the source character by character
  • Tracks (line, col) on every token
  • Skips whitespace and // or /* */ comments
  • Recognises keywords, identifiers, literals, operators

Token stream

INT      'int'      line 1 col 1
IDENT    'x'        line 1 col 5
ASSIGN   '='        line 1 col 7
INT_LIT  3          line 1 col 9
PLUS     '+'        line 1 col 11
INT_LIT  4          line 1 col 13
SEMI     ';'        line 1 col 14
EOF
Emphasise that the lexer is hand-written. The line/column tracking is what makes downstream errors useful.
Phase 2 · Parser

Tokens → Abstract Syntax Tree

Source

int x = 3 + 4 * 2;

How precedence works

  • Statements use recursive descent — one function per construct
  • Expressions use Pratt precedence climbing — a single binding-power table replaces N nested functions
  • * binds tighter than +, so 4 * 2 nests under +

AST (simplified)

Point at the AST. The shape encodes precedence — the multiplication is the right child of the addition because * binds tighter than +. Pratt parsing makes this almost free.
Phase 3 · Semantic Analyzer

Does the program make sense?

Broken source

int add(int a, int b) {
    return a + b;
}

int main() {
    x = 5;             // undeclared
    int y = 1;
    int y = 2;         // redeclared
    int z = add(1);    // wrong arity
    print(undeclared); // unknown name
    return 0;
}

Errors collected in one run

line 6:  'x' not declared
line 8:  'y' already declared in this scope
line 9:  'add' expects 2 arg(s), got 1
line 10: 'undeclared' not declared

Two passes

  • Pass 1: register every function signature & global so functions can call each other in any order
  • Pass 2: check each body — types, scope, arity, return type, array indices
Highlight that errors are collected and printed together — not fail-fast. The user sees the whole picture in one compile.

    {
        int x = 2;    // shadows outer x
        print(x);     // 2
    }

}

Scope stack at print(x) inner

Explain the stack-of-dictionaries model. Lookup walks from the top (innermost) downward, which gives variable shadowing for free.
Phase 4 · IR Generation

AST → Three-Address Code

Source: recursive factorial

int fact(int n) {
    if (n <= 1) {
        return 1;
    }
    return n * fact(n - 1);
}

int main() {
    print(fact(6));
    return 0;
}

Generated TAC

func_begin fact
param n
t1 = n <= 1
if_false t1 goto L1
return 1
L1:
t2 = n - 1
param t2
t3 = call fact, 1
t4 = n * t3
return t4
func_end fact

func_begin main
param 6
t5 = call fact, 1
print t5
return 0
func_end main
Walk through the TAC line by line. Notice the standard if/goto pattern, the param/call convention, and how the recursive call is just another function call.
Control Flow

How loops become labels & jumps

Source

while (i < 5) {
    s = s + i;
    i = i + 1;
}

Pattern

Ltop:
  ... evaluate cond ...
  if_false cond goto Lend
  ... loop body ...
  goto Ltop
Lend:

Lowered TAC

L1:
t1 = i < 5
if_false t1 goto L2
t2 = s + i
s = t2
t3 = i + 1
i = t3
goto L1
L2:

The same pattern lowers for loops too — they desugar to while with an init statement and an update at the end of the body.

This slide demystifies "where do labels and gotos come from?" — they fall out of the standard while/if lowering pattern.
Innovation · Arrays

Arrays end-to-end (bubble sort)

Source (excerpt)

int a[5];
a[0]=5; a[1]=2; a[2]=4; a[3]=1; a[4]=3;

while (i < 5) {
    int j = 0;
    while (j < 4 - i) {
        if (a[j] > a[j+1]) {
            int tmp = a[j];
            a[j] = a[j+1];
            a[j+1] = tmp;
        }
        j = j + 1;
    }
    i = i + 1;
}

TAC for the swap

t8  = a[j]
tmp = t8
t9  = j + 1
t10 = a[t9]
a[j] = t10
t11 = j + 1
a[t11] = tmp

Two new instructions

array_load: t = a[i] array_store: a[i] = v

The semantic analyzer enforces that the index is int and the name is actually an array.

Show that arrays touch every phase: lexer ([ ]), parser (ArrayDecl/ArrayAccess/ArrayAssign), semantic checks, and two new TAC ops.
Innovation · Optimizer

Constant folding (compile-time evaluation)

If both operands of a binary op are literals, evaluate them at compile time and replace the subtree with a single literal.

Source

int a = 2 + 3 * 4;
int b = (10 - 4) / 2;
int c = a + b;

Without optimization

t1 = 3 * 4
t2 = 2 + t1
a = t2
t3 = 10 - 4
t4 = t3 / 2
b = t4
t5 = a + b
c = t5

With optimization

a = 14
b = 3
t1 = a + b
c = t1

8 instructions → 4. Constant subtrees collapse before TAC is even generated.

Make the size difference concrete — 8 ops down to 4. The folding happens on the AST so all downstream phases benefit automatically.
Innovation · Optimizer

Dead code elimination (unreachable code)

After a return or goto, no instruction can run until the next reachable label. We drop everything in between, then drop labels nobody jumps to.

Source

int main() {
    int x = 5;
    return x;
    int y = 10;     // unreachable
    print(y);       // unreachable
    return 0;       // unreachable
}

Without optimization

func_begin main
x = 5
return x
y = 10
print y
return 0
func_end main

With optimization

func_begin main
x = 5
return x
func_end main
Two passes: collect all label targets, then sweep the code dropping unreachable instructions and unreferenced labels.

Honest scope

Limitations & future work

Out of scope today

  • Pointers, & and *
  • Strings, structs, enums, typedef
  • switch, break, continue, goto
  • Preprocessor & standard library headers
  • Native code generation — we stop at TAC

Natural next steps

  • Lower TAC to LLVM IR or x86-64 to produce executables
  • Add break / continue for loops
  • Common-subexpression elimination & copy propagation
  • A small register allocator after instruction selection
Be upfront about scope. The deadline drove these choices. Future work shows where the project would go next.
